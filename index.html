Compare

与普通项目页相比，Typex 更像一个正式产品官网

普通开源项目页

  • 通常只有一个 Demo，定位与价值表达较弱
  • 文档、API、体验页分散，用户路径容易中断
  • 更像代码入口，不像面向客户或团队的官网

Typex 官网首页

  • 首页负责解释定位、优势、场景与价值
  • 功能介绍页继续承接集成文档和 API 介绍
  • Demo 独立成一级页面，且拆成两个体验子标签

API

API 介绍

先理解公开方法与边界，再进入完整 API 页面查看明细。

1. 最小接入

import createEditor from 'editor'

const editor = createEditor({
  data: 'hello world',
})

editor
  .setToolBar(['undo', 'redo', 'header', 'color', 'bold', 'underline'])
  .mount('editor-root')

2. Release 交付内容

  • typex-editor-v2.0.min.js：浏览器推荐使用的压缩版
  • typex-editor-v2.0.js：浏览器调试更方便的标准版
  • typex-sdk-v2.0.zip：包含 editor / core / platform 三组产物

3. 核心接入流程

  • 先创建 editor 实例并挂载到容器
  • 再配置工具栏和 change 监听
  • 按需接入预览、文档输出和业务命令
  • 如果需要更深层扩展，再继续查看 API 介绍

4. 适合接入的方式

  • 浏览器 script 直接引入压缩版 SDK
  • 工程内通过包入口按模块接入
  • 把 Typex 作为业务编辑器底层能力封装

5. 继续阅读

  • 查看 v2.0 Release
  • 使用方式与调用流程
  • 目标架构

公开能力概览

  • mount(id)：挂载编辑器
  • setToolBar(items)：设置工具栏
  • getValue()：获取当前文档
  • setValue(data)：写入新内容
  • execCommand(name, value)：执行编辑命令

建议阅读顺序

先看集成文档，理解最小接入与交付方式；再看这里的 API 介绍，理解方法边界；最后再进入下方完整 API 页面按需查阅。

为什么 API 介绍要放在官网里

因为官网不仅服务“第一次认识产品的人”，也服务“马上要开始接入的人”。Typex 把这两类路径做在同一站点结构中，转化更完整。

Interactive Demo

Demo 演示

Demo 演示单独做成一级页面，并且拆成两个子标签：一个展示富文本编辑与预览，一个展示 Markdown 输入与预览。桌面端半屏编辑、半屏预览；移动端自动改为上下布局。

Rich Text Editor

富文本编辑

支持标题、字号、字体颜色、加粗、下划线、删除线等格式操作。

Live Preview

富文本预览

右侧同步渲染富文本内容，方便观察标题、颜色和文本格式效果。

Markdown Input

Markdown 输入

左侧输入 Markdown，右侧立即查看渲染结果，适合展示输出体验。

Live Preview

Markdown 预览

支持基础标题、段落、加粗、下划线、删除线和颜色渲染展示。

体验方式

子 Tab 演示

避免把不同类型演示混在一个版面里，让每种体验更清晰、更利于讲解。

桌面端

半屏编辑 / 半屏预览

更像真实产品 Demo 页面，适合产品经理、开发者和客户演示使用。

移动端

自动堆叠适配

在窄屏下自动改为上下结构，保证阅读和输入空间都足够清晰。
